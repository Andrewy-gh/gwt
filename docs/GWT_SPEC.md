# gwt - Git Worktree Manager

A CLI/TUI tool for managing Git worktrees with automated setup, file copying, and Docker Compose scaffolding.

## Overview

`gwt` simplifies the workflow of creating and managing Git worktrees by automating the tedious parts: copying configuration files, setting up Docker containers, installing dependencies, and running migrations.

## Core Features

### 1. Create Worktree

**Functionality:**
- Create a new worktree from a new or existing branch
- Support both local and remote branches
- Place worktrees in sibling directories (`../project-branch-name`)

**Modes:**
- **Interactive TUI** (default): Full keyboard-navigable interface
- **Non-interactive**: Flag-driven for scripting (`--branch`, `--no-tui`)

**Branch Sources:**
- New branch from current HEAD (default)
- Existing local branch
- Remote branch (fetches if needed)

**Validation:**
- Validate branch name format
- Check if worktree already exists
- Verify git repository state
- Prevent creation inside another worktree (warn, allow with `--force`)

**Branch Naming:**
- Slashes in branch names converted to hyphens for directory name
- Example: `feature/auth/login` → `../project-feature-auth-login`

**Remote Branch Handling:**
- Auto-create local tracking branch with same name
- If local branch exists with different ref, show error with options

**Directory Collision:**
- If target directory exists (not a worktree), prompt for alternative name

**Failure Handling:**
- On any failure, roll back completely (delete partial worktree, remove git reference)
- Clean state allows simple retry after fixing the issue

---

### 2. List Worktrees

**Display Information:**
- Branch name
- Filesystem path
- Clean/dirty status (uncommitted changes)
- Last commit message (truncated)
- Age (time since last commit)
- Linked remote branch (if any)

**Features:**
- Sortable columns
- Filter/search
- Batch selection for operations

---

### 3. Delete Worktrees

**Functionality:**
- Delete single or multiple worktrees (batch mode)
- Always prompt whether to delete the associated branch
- Clean up generated files (compose overrides, symlinks)

**Safety:**
- Warn if worktree has uncommitted changes
- Confirm before deletion
- Option to force delete (`--force`)
- Check if branch is merged into main/master
- Check if branch exists on remote (prompt for remote deletion)
- Prevent deletion of main worktree entirely
- Display warning: "This action cannot be undone"

**Pre-Deletion Checks:**
```
Deleting worktree: feature-auth

Pre-flight checks:
  ⚠ Worktree has uncommitted changes (3 files modified)
  ✓ Branch 'feature-auth' is merged into 'main'
  ⚠ Branch 'feature-auth' exists on remote 'origin'

⚠ Warning: This will permanently delete:
  • Worktree directory: /projects/myapp-feature-auth
  • All uncommitted changes in this worktree

This action cannot be undone.
```

---

### 4. File Copying

**Discovery Method:**
- Use `git status --ignored` to find all ignored files
- Parse output to get actual file paths (not just patterns)

**TUI Interface:**
- Checkbox list for file selection
- Pre-select files based on config
- Dependency directories unchecked by default:
  - `node_modules/`
  - `vendor/`
  - `.venv/`
  - `__pycache__/`
  - `target/` (Rust)
  - `dist/`
  - `build/`

**File Handling:**
- Copy files maintaining directory structure
- Support both files and directories
- Show copy progress for large directories

---

### 5. Docker Compose Scaffolding

**Auto-Detection:**
- Find all compose files:
  - `docker-compose.yml`
  - `docker-compose.yaml`
  - `docker-compose.*.yml`
  - `compose.yml`
  - `compose.yaml`
  - `compose.*.yml`

**Two Modes:**

#### Shared Mode (Default)
- Symlink data directories to main worktree
- All worktrees share the same container data
- No compose file modifications needed
- Use case: Shared database, cache, etc.

**Symlink Fallback (Windows):**
- Try symlink first
- If symlink fails (permissions), fall back to junction (directories only)
- If junction fails, fall back to copy
- Display message about enabling Developer Mode for symlinks

#### New Mode
- Copy data directories to new worktree
- Rename named volumes to avoid conflicts:
  - `postgres_data` → `postgres_data_feature-branch`
- Generate a compose override file (`docker-compose.worktree.yml`)
- Warn about port conflicts and suggest alternatives

**Override File Example:**
```yaml
# Auto-generated by gwt - do not edit
# To use: docker compose -f docker-compose.yml -f docker-compose.worktree.yml up

version: "3.8"

volumes:
  postgres_data_feature-branch:
    name: postgres_data_feature-branch

services:
  db:
    volumes:
      - postgres_data_feature-branch:/var/lib/postgresql/data
    ports:
      - "5433:5432"  # Changed from 5432 to avoid conflict
```

**Post-Generation Messaging:**
```
Created docker-compose.worktree.yml

To start containers for this worktree:
  docker compose -f docker-compose.yml -f docker-compose.worktree.yml up

Created ./dc helper script for convenience.
Note: If 'dc' conflicts with another command, rename it.
```

**Data Directory Paths:**
- Relative paths resolved from repository root
- Absolute paths supported but display warning about portability

### 6. Dependency Installation

**Auto-Detection:**

| File | Package Manager | Command |
|------|-----------------|---------|
| `package.json` + `bun.lockb` | Bun | `bun install` |
| `package.json` + `pnpm-lock.yaml` | pnpm | `pnpm install` |
| `package.json` + `yarn.lock` | Yarn | `yarn install` |
| `package.json` | npm | `npm install` |
| `go.mod` | Go | `go mod download` |
| `Cargo.toml` | Cargo | `cargo fetch` |
| `requirements.txt` | pip | `pip install -r requirements.txt` |
| `pyproject.toml` | Poetry/pip | Detect and run appropriate command |

**Features:**
- Detect multiple package managers in monorepo
- Run installations in parallel where possible
- Skip flag: `--skip-install`
- Show progress/output

---

### 7. Database Migrations

**Auto-Detection:**
- `Makefile` with `migrate-up` or `migrate` target
- `prisma/` directory → `npx prisma migrate deploy`
- `drizzle/` directory → `npx drizzle-kit migrate`
- `migrations/` with `.sql` files → Prompt for tool
- `alembic/` directory → `alembic upgrade head`

**Features:**
- Check if database container is running before attempting
- Skip flag: `--skip-migrations`
- Show migration output

## Configuration File

Location: `.worktree.yaml` in repository root (can be committed)

```yaml
# gwt configuration file

# Files and directories to pre-select for copying
# Supports glob patterns
copy_defaults:
  - ".env"
  - "**/.env"
  - "**/.env.local"
  - ".claude/"
  - "**/*.local.md"
  - "**/setenv.sh"

# Patterns to never select by default (even if gitignored)
copy_exclude:
  - "node_modules"
  - "vendor"
  - ".venv"
  - "__pycache__"
  - "target"
  - "dist"
  - "build"
  - "*.log"

# Docker configuration
docker:
  # Compose files to consider (auto-detected if not specified)
  compose_files:
    - "docker-compose.yml"
    - "docker-compose.dev.yml"

  # Data directories that should be symlinked (shared) or copied (new)
  data_directories:
    - "server/db-data"
    - "data/redis"

  # Default mode: "shared" or "new"
  default_mode: "shared"

  # Port offset for new containers (e.g., 5432 -> 5433)
  port_offset: 1

# Dependency installation settings
dependencies:
  # Auto-detect and install (default: true)
  auto_install: true

  # Directories to check for package managers
  paths:
    - "."
    - "client"
    - "server"
    - "packages/*"

# Migration settings
migrations:
  # Auto-detect and offer to run (default: true)
  auto_detect: true

  # Custom migration command (overrides auto-detection)
  # command: "make migrate-up"

# Post-setup hooks (run after everything else)
# Hooks run in the new worktree directory with GWT_* environment variables
hooks:
  post_create:
    - "echo 'Worktree ready!'"

  post_delete:
    - "echo 'Worktree removed'"
```

**Hook Environment Variables:**

| Variable | Example | Description |
|----------|---------|-------------|
| `GWT_WORKTREE_PATH` | `/projects/myapp-feature-x` | New worktree absolute path |
| `GWT_BRANCH` | `feature-x` | Branch name |
| `GWT_MAIN_WORKTREE` | `/projects/myapp` | Main worktree path |
| `GWT_SOURCE_REF` | `main` or `origin/feature-x` | What branch was created from |

**Hook Execution:**
- Hooks run in the new worktree directory
- Environment variables available for scripting
- Example: `cp "$GWT_MAIN_WORKTREE/.env.template" .env`

## CLI Interface

### Commands

```bash
# Launch interactive TUI (default)
gwt

# Create worktree
gwt create                          # Interactive mode
gwt create -b feature-login         # New branch, non-interactive
gwt create -b feature-login --from main  # New branch from specific ref
gwt create --checkout existing-branch    # Existing branch
gwt create -b feature --skip-install --skip-migrations
gwt create -b feature --copy-config  # Copy .worktree.yaml to new worktree

# List worktrees
gwt list                            # Interactive list with details
gwt list --json                     # JSON output for scripting
gwt list --simple                   # Just paths, one per line

# Delete worktrees
gwt delete                          # Interactive selection
gwt delete feature-branch           # Delete specific worktree
gwt delete feature-1 feature-2      # Delete multiple
gwt delete --all                    # Delete all (with confirmation)
gwt delete feature --delete-branch  # Also delete the branch
gwt delete feature --force          # Skip confirmations

# Utility commands
gwt status                          # Show current worktree info
gwt config                          # Edit/view configuration
gwt config init                     # Create default .worktree.yaml
gwt doctor                          # Check prerequisites and system health
```

**gwt doctor Output:**

## Design Decisions

### Windows Symlinks
**Decision:** Fall back to junction, then copy.
- Try symlink first
- On Windows, fall back to junction (directories only)
- Final fallback: copy with warning
- Display message about enabling Developer Mode for symlink permissions

### Git Library
**Decision:** Shell out to `git` CLI.
- Fewer edge cases than go-git
- User's git config (credentials, hooks) honored automatically
- Better worktree support
- Requires git to be installed (validated by `gwt doctor`)

### Compose File Generation
**Decision:** Generate override file.
- Non-destructive (original compose file untouched)
- Easy to gitignore (`docker-compose.worktree.yml`)
- Version control serves as backup
- Generate `dc` helper script for convenience

### Multi-Repo Support
**Decision:** Out of scope for v1.
- Single-repo focus per `gwt` invocation
- Mono-repo support via `dependencies.paths` config
- Multi-repo orchestration could be future enhancement

### Configuration Inheritance
**Decision:** Read from main worktree by default.
- Worktrees read `.worktree.yaml` from main worktree
- Optional `--copy-config` flag to copy config to new worktree
- Ensures consistent behavior across worktrees

### Bare Repository Support
**Decision:** Not supported in v1.
- Display clear error message if detected
- Typical users have standard clones

### Concurrent Operations
**Decision:** Use lock file.
- Prevent concurrent `gwt create` operations
- Lock file stored in `.git` directory
- Clear error message if another process is running
